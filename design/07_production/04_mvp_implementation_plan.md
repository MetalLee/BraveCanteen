# MVP 实现方案

状态：草案  
上级索引：[[design/README|Design Knowledge Base]]

## 目的

本文档把现有设计整理成首版可开发方案，用于连接 [[design/00_product/00_game_concept|游戏概述]]、[[design/01_core_gameplay/00_core_loop|核心循环]]、[[design/00_product/03_scope_and_success_criteria|项目范围与成功标准]] 和 Godot 实现任务。

首版目标不是做完整商业版本，而是验证玩家能否在短时间内理解并享受：

```text
金币 -> 食材 -> 菜品 -> 勇者属性 -> 魔物讨伐 -> 奖励 -> 餐厅成长
```

## 首版体验定义

玩家经营一家服务勇者的幻想餐厅。餐厅不模拟复杂厨房动线，而是作为勇者成长机器存在：玩家购买食材，制作属性菜品，喂给勇者提升生命、攻击或防御；魔物袭击倒计时持续施压，玩家在倒计时前选择队伍派遣讨伐；自动战斗根据队伍和魔物攻防数值结算；胜利奖励反哺餐厅、食谱、战利品和下一轮成长。

## MVP 玩家流程

1. 进入主界面，看到金币、库存、勇者、当前袭击倒计时。
2. 花费金币购买基础食材。
3. 使用食材制作菜品。
4. 将菜品喂给勇者，提升对应属性。
5. 查看袭击预警，理解魔物威胁方向和可能奖励。
6. 根据预测标签选择出战勇者。
7. 自动结算战斗，显示关键攻防差距、回合数和胜负原因。
8. 成功后领取金币、食材、食谱或战利品。
9. 使用奖励升级餐厅或解锁新内容。
10. 进入下一轮更高效率、更高威胁的循环。

## 首版界面

### 餐厅总览

用途：让玩家快速判断当前最值得做的 1-3 个动作。

必须展示：

- 金币。
- 餐厅等级。
- 袭击倒计时。
- 当前魔物名称、威胁等级和预测标签。
- 快捷入口：食材、烹饪、勇者、派遣、升级。

### 食材与烹饪

用途：完成金币到菜品的转换。

必须支持：

- 购买基础食材。
- 查看食材库存。
- 查看可制作菜品。
- 显示菜品消耗和属性效果。
- 制作菜品并加入库存。

### 勇者

用途：完成菜品到战力的转换。

必须支持：

- 查看勇者生命、攻击、防御。
- 查看职业定位。
- 给勇者使用菜品。
- 显示属性提升结果。
- 标记勇者是否可出战。

### 袭击与派遣

用途：让玩家理解威胁、做准备并派遣队伍。

必须支持：

- 显示剩余时间。
- 显示魔物生命、攻击、防御和威胁标签。
- 选择多名勇者。
- 显示战斗预测标签：稳定胜利、有风险、极危险、几乎无法取胜。
- 开始讨伐。

### 战斗结果

用途：解释胜负并强化奖励反馈。

必须展示：

- 出战勇者列表。
- 队伍总生命、攻击、防御。
- 魔物生命、攻击、防御。
- 双方有效伤害。
- 击败所需回合数。
- 胜负结果。
- 奖励列表。

## 首版规则

### 经济

- 初始金币：60。
- 初始可购买食材：小麦、蘑菇、肉块、青菜。
- 初始勇者：莉娜、布拉姆。
- 第二场战斗后解锁：弓箭手米拉、游侠塞莉娅。
- 初始出战人数上限：2。
- 餐厅等级 2 解锁香草、欧文和一个技能节点。

### 自动战斗

首版沿用 [[design/04_balance_data/02_attribute_and_combat_formula|属性与战斗公式]]：

```text
team_hp = sum(adventurer.hp)
team_attack = sum(adventurer.attack)
team_defense = sum(adventurer.defense)

team_damage_per_round = max(1, team_attack - monster.defense)
monster_damage_per_round = max(1, monster.attack - team_defense)

rounds_to_defeat_monster = ceil(monster.hp / team_damage_per_round)
rounds_to_defeat_team = ceil(team_hp / monster_damage_per_round)

success = rounds_to_defeat_monster <= rounds_to_defeat_team
```

### 失败处理

首版采用温和方案：

- 不获得讨伐奖励。
- 出战勇者进入短暂休整。
- 不损失餐厅等级、食谱或战利品。

这样可以优先验证核心闭环，避免早期失败破坏玩家继续调整的动机。

## 技术拆分

规则层模块：

- `Economy`：金币、食材库存、菜品库存。
- `Cooking`：食谱、食材消耗、菜品效果。
- `Adventurer`：勇者属性、喂食成长、出战状态。
- `Raid`：袭击倒计时、当前魔物、派遣队伍。
- `AutoBattle`：战斗预测和结算。
- `Reward`：奖励表、掉落、战利品被动。
- `Progression`：餐厅等级、解锁条件、技能节点。
- `Save`：本地存档。

表现层原则：

- Godot 场景只读取规则层状态和事件日志。
- UI 不直接结算金币、掉落或战斗。
- 数据表优先驱动食材、菜品、勇者、魔物和战利品。

## 开发顺序

1. 建立数据表：食材、菜品、勇者、魔物、战利品。
2. 编写无场景依赖的规则层原型。
3. 实现自动战斗预测和结算测试。
4. 实现购买食材、制作菜品、喂养勇者。
5. 实现袭击倒计时和派遣。
6. 实现奖励领取和战利品被动。
7. 创建 Godot 主界面和四个核心面板。
8. 接入存档。
9. 进行 20 分钟内完整循环测试。

## 验收标准

- 玩家 3 分钟内能理解做菜强化勇者、派勇者讨伐、奖励反哺餐厅。
- 第一次完整循环在 5-8 分钟内完成。
- 战斗结果页能解释胜负原因。
- 至少包含 6 种食材、6 种菜品、4 名勇者、4 种魔物、4 个战利品。
- 自动战斗和资源结算可在无 Godot 场景依赖下测试。
- Godot 原型可完成从购买到讨伐奖励的完整闭环。

## 待验证问题

- 首次袭击倒计时 180 秒是否足够玩家理解购买、烹饪和喂食。
- 初始出战人数上限 2 是否会让队伍选择足够有意义。
- 餐厅等级 2 的解锁是否应该优先给新勇者、新食材，还是技能树节点。
- 失败休整时长是否能形成压力，又不会打断继续游玩。
- 战利品被动加成是否需要统一采用加法叠加，避免倍率失控。
